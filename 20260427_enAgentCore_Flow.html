3-Layer Permission Structure:

  • ① User Authentication: Cognito + IDP (who is accessing)
  • ② Service-to-Service Authorization: IAM Role (what each AWS service can do)
  • ③ Cross-Account: sts:AssumeRole (operating resources in other accounts)

① User Authentication — Cognito + IDP Configuration

Authentication flow when a user accesses from a web browser. Achieves SSO by integrating with IDP.

Cognito User Pool Settings

  • IDP federation configuration
  • User group: mode2-ops-users
  • MFA: Required (OTP email)
  • Token expiry: 1 hour

Cognito Identity Pool Settings

  • Assigns IAM role to authenticated users
  • Authenticated Role: mode2-cognito-auth-role
  • Only API calls via ALB are permitted
// IAM Role for Cognito Authenticated Users (Least Privilege)
{
  "Effect": "Allow",
  "Action": [
    "execute-api:Invoke"  // Only API calls via ALB
  ],
  "Resource": "arn:aws:execute-api:ap-northeast-1:COMMON_ACCOUNT:*"
}

OTP Two-Factor Authentication Flow:
Cognito Auth → Portal Login → Lambda(login_auth) → DynamoDB Save → SES Email → User enters OTP → Lambda(validate) → Access Granted

② ECS Task Role — Fargate Container Permissions

Permissions for ECS/Fargate containers to call AWS services. This is the "core" permission of the Chatbot.

// ECS Task Role: mode2-ecs-task-role
// Trust Policy (allows ECS tasks to assume this role)
{
  "Principal": { "Service": "ecs-tasks.amazonaws.com" },
  "Action": "sts:AssumeRole"
}

// Permission Policy
{
  "Statement": [
    // 1. Bedrock invocation
    { "Action": ["bedrock:InvokeModel", "bedrock:InvokeModelWithResponseStream"],
      "Resource": "arn:aws:bedrock:ap-northeast-1::foundation-model/*" },

    // 2. AgentCore operations
    { "Action": "bedrock-agentcore:*", "Resource": "*" },

    // 3. DynamoDB (OTP management)
    { "Action": ["dynamodb:PutItem", "dynamodb:GetItem", "dynamodb:UpdateItem", "dynamodb:Query"],
      "Resource": "arn:aws:dynamodb:ap-northeast-1:*:table/ChatBot-OTP-Verification" },

    // 4. Secrets Manager (retrieve credentials)
    { "Action": "secretsmanager:GetSecretValue",
      "Resource": "arn:aws:secretsmanager:ap-northeast-1:*:secret:mode2-*" },

    // 5. Cross-account AssumeRole (most critical)
    { "Action": "sts:AssumeRole",
      "Resource": "arn:aws:iam::*:role/mode2-ops-cross-account-role" }
  ]
}
✅ Allowed — Bedrock
  • Claude Sonnet / Haiku
  • Titan Embeddings
  • Knowledge Base search
✅ Allowed — AgentCore
  • Runtime invocation
  • Memory read/write
  • Observability
❌ Denied — Direct EC2 Operations
  • Direct EC2 operations not allowed
  • Must go through AssumeRole
  • (for audit trail)

Common Account Side (caller)

// Policy attached to ECS Task Role
{
  "Effect": "Allow",
  "Action": "sts:AssumeRole",
  "Resource": [
    "arn:aws:iam::B2C_NONPROD_ID:role/mode2-ops-role",
    "arn:aws:iam::B2C_PROD_ID:role/mode2-ops-role",
    "arn:aws:iam::INTERNAL_NONPROD_ID:role/mode2-ops-role",
    "arn:aws:iam::INTERNAL_PROD_ID:role/mode2-ops-role",
    "arn:aws:iam::GAMMA_NONPROD_ID:role/mode2-ops-role",
    "arn:aws:iam::GAMMA_PROD_ID:role/mode2-ops-role"
  ]
}

Each Target Account Side (receiver)

// mode2-ops-role Trust Policy
{
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::COMMON_ACCOUNT_ID:role/mode2-ecs-task-role"
  },
  "Action": "sts:AssumeRole",
  "Condition": {
    "StringEquals": {
      "sts:ExternalId": "mode2-chatbot-trusted"
    }
  }
}

// mode2-ops-role Permission Policy (created in each target account)
{
  "Statement": [
    // EC2 operations (start, stop, reboot, status check)
    { "Action": [
      "ec2:StartInstances", "ec2:StopInstances", "ec2:RebootInstances",
      "ec2:DescribeInstances", "ec2:DescribeInstanceStatus"
    ], "Resource": "*",
    // Tag condition: only instances with mode2-managed=true can be operated
    "Condition": { "StringEquals": { "ec2:ResourceTag/mode2-managed": "true" } } },

    // RDS operations (start, stop, status check)
    { "Action": [
      "rds:StartDBInstance", "rds:StopDBInstance",
      "rds:DescribeDBInstances", "rds:DescribeDBClusters"
    ], "Resource": "*" },

    // CloudWatch (logs and metrics)
    { "Action": ["cloudwatch:GetMetricData", "logs:GetLogEvents", "logs:DescribeLogGroups"],
      "Resource": "*" }
  ]
}

Why use ExternalId:
To prevent the "Confused Deputy Problem." Without ExternalId, a malicious third party who knows the Common Account role ARN could attempt AssumeRole. Setting ExternalId ensures only legitimate Chatbot calls are accepted.

⑤ OTP Authentication Flow — Two-Factor Auth for EC2 Operations

Before performing dangerous operations such as starting or stopping EC2, identity is verified via OTP.

1
User requests EC2 operation
User asks Chatbot to stop EC2, etc. → Strands Agent calls OTP generation Lambda
2
Lambda: Generate OTP → Save to DynamoDB
Generates 6-digit code → Saves to ChatBot-OTP-Verification table (TTL: 10 min)
⚠ Note: Current code outputs OTP to CloudWatch Logs — must be removed before production
3
Send OTP email via SES
Sends 6-digit code to user's email address
Sender: no-reply-security@mail.nissan.co.jp
4
User enters OTP → Lambda: Validation
Verifies against DynamoDB (expiry check, used-flag check) → EC2 operation permitted on success
5
Connect to target account via sts:AssumeRole → Execute EC2 operation
Obtains temporary credentials (15 min–1 hour) to operate EC2
After operation, sends completion notification email to team and user

Principle of Least Privilege:

  • ECS Task Role cannot directly operate EC2 → must go through AssumeRole (audit trail is preserved)
  • mode2-ops-role can only operate resources tagged with mode2-managed=true
  • OTP Lambda can only access the specific DynamoDB table
  • All operations are recorded in CloudTrail
